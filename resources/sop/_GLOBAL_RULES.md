# 全局规则（所有 SOP 卡片都必须遵守）

## 规则 1：「继续」歧义消除（最高优先级）
"继续" / "嗯" / "好" / "下一个"单独出现时，永远不代表 `[下一内容]`。

| 用户输入 | 上下文 | AI 解读 | AI 行为 |
|---------|-------|--------|---------|
| "继续" / "嗯" / "好" / "下一个" | 当前单元讲解中 | 当前单元内继续讲解 / 等待用户回应 | 继续当前段落，**禁止**切换学习单元 |
| "继续" / "继续走" / "接着来" | API 错误 / 工具失败 / 流程被打断后 | 继续被中断的当前流程（同一 Step / 单元） | 从中断点恢复执行，禁止跳到新 Step / 单元 |
| `[下一内容]` / `[下一内容]` 对应短语 | 任意 | 推进下一单元 | 走 `SOP_下一内容.md` 流程 |
| `[超前学习]` | 单元已全部完成 | 提前加载明日的首个单元 | 走 `SOP_下一内容.md` 超前分支 |
| 模糊（如"OK 走吧"） | 任意 | 必须先反问澄清 | 输出歧义消除模板（见下） |

### 歧义消除模板（必输，原文照搬）
```
你刚才说的「<用户原话>」我无法确定意图。请二选一：
1. 继续当前单元（我刚才讲到 XXX，接着往下讲）
2. [下一内容]（我会先输出「掌握情况检查」并请你 [同步]）
```

## 规则 2：每个学习单元结束必输「掌握情况检查」
触发条件：AI 判定讲解完毕、用户主动说 `[下一内容]`、或对话到 5-6 轮。

### 必输模板（原文照搬）
```
---【掌握情况检查】---
当前学习单元：<单元名>
- 已讲解知识点：<A、B、C 列表>
- 用户提问/卡点：<D（已解答）、E（待巩固）>
- 掌握度评估：[已掌握 / 基本掌握 / 需巩固]
- 编码进度：[已完成 / 进行中 / 未开始 / 不适用]

请确认：
1. 继续下一内容（说 [下一内容]）
2. 再巩固一下当前内容
3. 我有新问题要问
4. 开始写代码（说 [开始写代码]）
---
```

## 规则 3：StudyFlow.md 不是参考，是强制
* "AI 主动询问同步" → **必须**询问。
* "推荐 1-2 篇业内论文" → **必须**推荐（Day 1 除外）。
* "AI 不直接给完整实现" → **禁止**单次连续输出超过 20 行代码（Day 21 CRUD/前端除外）。

## 规则 4：JSON 状态机与状态自动保存（防漂移与自愈核心）
本系统以 `docx/StudyState.json` 为单一事实源。

### 状态自动保存触发时机
* 完成一个学习单元 / 单元确认推进后
* 任何 `[同步] XXX` 触发时
* 今日学习复盘评分 / 结束学习时

### 单元状态枚举（统一定义，禁止新增取值）
`StudyState.json` 中 `units[].status` 只允许以下 4 种取值：
* `"not_started"` — 未开始
* `"in_progress"` — 讲解中（含 `[超前学习]` 拽入的单元，可附加 `"ahead": true` 标记）
* `"completed"` — 已完成（必须同时有 `rating` 实际分值）
* `"postponed"` — 顺延至次日（`[结束今日学习]` 时对未完单元设置）

### JSON 读写与自愈守则（强制）
1. **读前置**：在修改状态前，**必须**先 Read `docx/StudyState.json`。
2. **写覆盖**：修改状态数据后，**必须**用 Write 覆写更新 `docx/StudyState.json`，并执行 JSON 语法校验，防格式损坏。
3. **Markdown 同步**：JSON 更新后，同步 Edit 对应的 `StudyMemory/Day_N.md` 与 `Study.md`。
4. **状态自愈（WAL 机制）**：若检测到 `StudyState.json` 丢失或损坏导致不可读，AI **必须**扫描 `docx/StudyMemory/` 目录（**仅**匹配 `Day_<NN>.md` 命名格式的文件，忽略 `archive/` 子目录及其他命名），读取编号 NN 最大的 `Day_<NN>.md`（作为预写日志 WAL）。解析其中的“今日导学单元”、“同步记录”、“进度”等 Markdown 信息，重新组装并自动恢复 `docx/StudyState.json` 后再继续，禁止直接 FAIL 崩溃。

## 规则 5：每个 SOP 执行前自检（FAIL-FAST）
SOP 卡片的「FAIL-FAST 自检」为强前置门禁。任一项为否，必须 **STOP**，输出未通过项并等待用户补充，禁止绕过直接执行。

---

## 规则 6：任何触发指令前置 Read
新会话收到指令后，第一个工具调用按顺序读：
1. `docx/StudyState.json` (若损坏则先执行规则 4 的状态自愈)
2. `docx/SOP/_GLOBAL_RULES.md`（本文件）
3. `docx/SOP/SOP_<指令名>.md`（若为跳转天数，则读 SOP_跳转天数.md）
禁止仅凭记忆或跳过读取直接执行。

---

## 规则 7：Markdown 标记字符 = 必输（适用所有 SOP 所有模板）
模板中出现的任何 Markdown 标记（`##`, `**`, `[ ]`, `---`, `|`, `>`）均为字面内容的一部分。
* **用户面输出 ≡ 文件 Edit**：无论是在终端输出给用户，还是 Edit 写入文件，Markdown 标记**必须完全保留**，禁止为美观将其剥离或替换。
* **回显检查**：在思考块内比对输出，若有缺失立即重新生成。

---

## 规则 8：SOP 模板与 StudyFlow.md 冲突时以 SOP 为准
当两者模板字段或字符不一致时，永远以 SOP 模板为准。禁止以 StudyFlow 为由省略 SOP 字段。

---

## 规则 9：用户面输出 ≡ 文件 Edit（禁止双标与自检欺骗）
1. **用户面输出与文件 Edit 一视同仁**：SOP 模板中的 `##` `**` `[ ]` `---` `|` `>` 等 Markdown 标记字符 = 字面内容。**禁止**在直接给用户的 assistant message 中剥离这些标记。Edit 文件时保留 ✅ 但给用户输出剥离 ❌ = **违规**。
2. **禁止自检欺骗**：内心 checklist 勾选 "✅" 不等于真的做了。SOP 模板输出后必须**逐字符**与原文 diff，发现 `##`/`**`/`[ ]`/`---` 缺失立即重新生成。

---

## 规则 10：模板字面锁定原则（Template Literal Lock）
SOP 模板 of 输出只能是模板原文字符串的精确副本（仅 `<占位符>` 内可替换）。禁止语义等价改写、禁止合并子块、禁止将列表改成段落。

---

## 规则 11：深度讲解黄金法则（Why, Where, QA 三要素）
为保证技术讲解深度，AI 讲解任何核心技术点时，**必须**同时包含以下三大板块（注：此深度讲解必须放在用户在 replica 项目中编写并成功跑通当前单元对应的标准简易 Demo 之后进行，严禁在前置的 Demo 编码阶段直接展示或分析原项目复杂的底层管道源码）：

1. **Why (架构决策与权衡)**：
   * 解释该设计的深层原因和替代方案对比。
   * 例如：“为什么选择 Pgvector 而不是 Milvus？因为当前项目数据量级在百万级以内，用 Pgvector 可以减少运维成本，实现关系数据与向量数据的强一致性，且避免分布式检索的网关开销。”
2. **Where (源码精准定位)**：
   * 必须给出原项目对应的具体类名、方法名，必要时指出行号（行号可基于 Project.md 和实际 Read 源码校验）。
   * 例如：“该熔断器逻辑实现在 `infra-ai` 模块的 `ModelHealthStore` 类中，熔断阈值判定主要在 `checkCircuitBreaker` 方法。”
3. **QA (面试三连问预案)**：
   * 提供该技术点在面试中的高频问法、30秒精炼回答、2分钟展开叙述以及追问应对（包含至少 2 个追问应对）。
   * 杜绝任何浅尝辄止的概念朗读，讲透底层机制。

## 规则 12：Day 21 代码行数特赦豁免
* 当 `current_day` 为 21（例外日：CRUD + 前端）时，AI 可以直接生成完整 CRUD 接口与 React 前端页面，**特赦豁免**单次代码 ≤ 20 行的硬性限制。
* 当 `current_day` 为其他数值时，必须严格拦截超过 20 行的连续代码输出。

## 规则 13：掌握度评估的真实性、多维度连环追问与分级量化（禁止自嗨式判断）
1. **多维度连环追问机制**：AI 对每个学习单元的掌握度评估，**禁止**仅凭单次问答或关键词匹配进行草率判定。AI **必须**围绕单元内的多个核心知识点，发起至少 **2 回合（连环追问）** 的真实问答校验。追问必须直击底层逻辑，例如：
   - 第一轮：基础概念/参数意义（What/How）；
   - 第二轮：底层数学逻辑、源码实现定位、业务架构决策（Why/Where）。
2. **掌握度分级量化（1.0 - 5.0分）**：AI 在「掌握情况检查」块中的掌握度评估**必须**废除主观定性词，统一采用量化分值，并严格遵循以下判定依据：
   - **`[5.0分 (优秀)]`**：能独立准确推导底层原理/数学公式，清晰说明 Ragent 项目中的源码类名、方法名，并能说明具体的架构权衡决策（Why & Where & QA 三要素齐备）。
   - **`[4.0分 (良好)]`**：能清晰说清技术原理与数学逻辑，对概念有深度认知，但源码行或细节记忆有轻微瑕疵。
   - **`[3.0分 (及格)]`**：仅能答出基本概念名词、表面接口流程，缺乏底层机制推导或源码认识。
   - **`[低于3.0分 (薄弱)]`**：答题错误、逻辑混淆，或未回答追问。
3. **默认“需巩固”原则**：在单元讲解结束且尚未通过上述“连环追问”考核前，AI 在「掌握情况检查」块中的评估**必须**写为 `[需巩固]`，且**必须**在后续交互中开始提问，**禁止**直接走 `SOP_下一内容.md` 的推进分支。
   * **与第 2 条的关系（澄清，不冲突）**：「掌握情况检查」块的评估栏在考核完成**前**填定性词 `[需巩固]`（本条款）；考核完成**后**的终期评分（`SOP_下一内容.md` Step 3 输出、StudyMemory「掌握度评分」栏、JSON `rating`）**必须**使用第 2 条的量化分值（如 `3.5分`）。禁止在终期评分栏继续填定性词。
4. **互动式带读与五步导学循环**：AI 导学每个学习单元时，必须严格执行以下 5 步流程，不得跳过或颠倒顺序（单次回复正文控制在 1000 字以内，以互动事实为唯一评判依据）：
   - **步骤一：文档带读**：AI 提炼当天文档/知识的核心要点。
   - **步骤二：Replica 交互式编码**：AI 在 `<复现名>` 下提供 Demo 骨架和关键逻辑占位（如 `// TODO` 填空），引导用户手写关键行并跑通调试，观察控制台输入输出。AI **禁止**直接给出完整可复制实现。
   - **步骤三：源码深度对照**：用户跑通后，AI 引导阅读 `<项目名>` 源码，深度讲解 Why/Where 架构设计与面试 QA。
   - **步骤四：前沿论文/文章带读**：推荐并精读 1 篇与当天内容相关的业内最新专业论文核心章节。
   - **步骤五：掌握度评估**：执行 2 回合连环拷问并进行分级量化打分。

## 规则 14：状态变更强制执行验证 Hook（备份 → 写入 → 校验 → 失败回滚）
1. **写入前备份**：AI 在执行任何会修改 `StudyState.json` 和 `Day_N.md` 中单元进度的操作（例如 `[下一内容]` 的状态落盘）前，**必须**先将目标文件复制备份到 `docx/hooks/backup/`（同文件名加 `.bak` 后缀，如 `StudyState.json.bak`、`Day_02.md.bak`）。
2. **写入后立即校验**：文件落盘**后**，AI **必须**立即运行校验命令：
   `python docx/hooks/validate_study.py`
3. **断言拦截与回滚**：若校验命令返回非零（即校验失败），AI **必须**立即终止当前流程，用 `docx/hooks/backup/` 中的备份恢复被修改的文件，并将具体的错误输出报告给用户，禁止在未通过校验的情况下宣布单元完成或切换至下一单元。
